# LiveYukiL2D 长期开发计划

## 1. 项目目标

LiveYukiL2D 的目标是构建一个单用户、单会话、单 LLM 与单 Live2D 模型绑定的桌面智能体。系统以 Yuki 为核心人格，通过聊天、主动对话、视觉理解、Live2D 表现和受控的电脑操作形成完整闭环。

核心约束：

- 使用单管线，不按 `chat_id` 拆分会话
- 一个运行实例对应一个 LLM、一个 Yuki 人格和一个 Live2D 模型
- 优先复用成熟模块和现有代码，不复制完整 React 应用
- 当前优先实现文本聊天、主动行为、视觉理解和状态展示
- GPT-SoVITS + RVC 属于后续媒体增强，不阻塞当前架构
- 电脑操作能力必须具备明确权限边界、用户确认和完整审计

## 2. 目标架构

```text
Electron 桌面窗口
        │
        ▼
Live2D 前端与状态界面
        │ WebSocket / HTTP
        ▼
LiveYuki Runtime
  ├─ 单管线会话调度器
  ├─ Yuki 主 Agent
  ├─ 执行子 Agent
  ├─ LLM 适配器
  ├─ 视觉理解适配器
  ├─ 屏幕与浏览器观察器
  ├─ 工具注册中心
  ├─ 记忆与状态系统
  └─ Live2D 事件输出器
```

单管线内部保持统一顺序：

```text
输入收集
  → 状态更新
  → 可选视觉观察
  → 上下文构建
  → Yuki 决策
  → 工具调用或直接回复
  → 文本/表情/动作输出
  → 状态和记忆保存
```

不引入 YukiV6 的多 `chat_id` 会话模型，但保留其防抖、串行执行、工具调用期间消息处理和阶段化管道思想。

## 3. Agent 角色决策

### 3.1 推荐方案：Yuki 主 Agent + 执行子 Agent

Yuki 作为唯一对外人格和最终决策者，负责：

- 理解用户输入
- 判断是否需要回复
- 决定是否主动发起对话
- 决定是否观察屏幕或浏览器
- 决定是否调用电脑操作能力
- 选择工具和任务优先级
- 将执行结果转化为符合 Yuki 人格的回复
- 决定 Live2D 表情、动作和状态

执行子 Agent 负责：

- 分解具体任务
- 查询文件、网页或系统信息
- 调用电脑操作工具
- 生成临时脚本或复用技能
- 返回结构化结果和操作记录

执行子 Agent 不直接决定对用户说什么，也不绕过 Yuki 直接发送对外消息。

### 3.2 不推荐的方案：让执行 Agent 取代 Yuki

如果电脑操作 Agent 直接成为完整主 Agent，会产生以下问题：

- 人格逻辑和执行逻辑混在一起
- 高权限工具容易被普通对话误触发
- 工具结果可能绕过统一的状态和记忆处理
- Live2D 表现难以和任务阶段保持一致
- 后续切换模型或人格时耦合较重

因此，执行 Agent 应作为 Yuki 的工具或下属能力，而不是第二个对外人格。

### 3.3 人格注入方式

执行子 Agent 可以注入 Yuki 的有限人格参数：

- 任务解释风格
- 风险偏好
- 对主人意图的优先级判断
- 汇报语气
- 是否主动询问澄清

但权限、工具白名单、确认策略和资源限制必须由运行时强制控制，不能交给人格提示词决定。

## 4. 功能路线

### 阶段一：稳定单管线基础

目标是把当前能运行的 Live2D 桌面宠物升级为可扩展运行时。

工作内容：

- 保留当前 `server.py`、Electron 和 minimal 前端的稳定启动链路
- 将 WebSocket 消息处理从 `main.ts` 中逐步抽出为协议路由层
- 统一 `text-input`、`say`、`audio`、`expression`、`set-model-and-conf` 消息
- 建立单管线状态对象
- 实现输入队列、防抖和同一时刻只执行一个推理任务
- 保留任务取消和中断能力
- 定义结构化输出事件：文本、音频、表情、动作、状态、工具事件
- 增加基础日志、错误事件和运行状态接口

完成标准：

- 聊天框输入可以经过后端 LLM 返回 Yuki 回复
- 同一时间不会并发运行多个主推理任务
- 回复可以驱动字幕、动作和表情
- 原有 `/api/say`、`/api/audio`、`/ws` 接口保持兼容

### 阶段二：聊天与 Yuki 人格

工作内容：

- 增加桌面聊天框和消息列表
- 接入 YukiV6 的上下文构建、历史管理和 LLM 客户端思想
- 将 YukiV6 的 `SessionPipeline` 改造成单管线版本
- 实现短期对话记忆
- 增加系统提示词、人格配置和运行参数配置
- 支持用户主动输入、清空会话、重置上下文和中断回复
- 统一 LLM 流式输出与 Live2D 输出事件

暂不迁移：

- QQ/NapCat 网络层
- 群聊成员和群聊状态
- YukiV6 的按 `chat_id` 状态表

### 阶段三：主动对话和灵动待机

工作内容：

- 建立用户活跃状态：输入时间、窗口状态、鼠标活动和最近交互
- 建立用户在线判断：窗口可用性、前端心跳、系统空闲状态等
- 实现主动对话策略，避免固定间隔机械发言
- 增加冷场、等待、注意到用户、任务完成等行为状态
- 为不同状态配置 Live2D 动作、表情、视线和字幕
- 扩展待机动作：呼吸、眨眼、视线游移、轻微姿态变化、随机动作组
- 将待机行为和 Yuki 的内部状态绑定，而不是只使用随机动作

主动对话必须具备：

- 总开关
- 安静时段
- 最短间隔
- 用户活跃度判断
- 用户明确拒绝后的冷却时间
- 生成失败时的退避策略

### 阶段四：视觉理解和主动窥屏

工作内容：

- 抽象统一的 `VisionProvider` 接口
- 支持截图输入、窗口截图和选区截图
- 接入云端视觉大模型
- 对截图进行压缩、裁剪、脱敏和频率限制
- 将视觉观察结果作为短期上下文，不默认写入长期记忆
- 支持用户手动请求“看看屏幕”
- 增加主动观察策略，由 Yuki 判断何时值得观察
- 记录观察原因、时间、输入范围和模型结果

安全要求：

- 默认关闭主动窥屏
- 首次启用时明确显示状态
- 支持暂停和立即停止
- 不上传密码框、隐私窗口和用户排除区域
- 云端调用前显示或记录上传范围
- API Key 只从环境变量或本地安全配置读取

### 阶段五：浏览器窥屏与网页 Agent

参考：

- `reference/YukiV6/future addons/webpage_agent/yuki_brain.py`
- `reference/YukiV6/future addons/webpage_agent/yuki_eye.user.js`

迁移方向：

- 将 HTTP 轮询原型改造成 WebSocket 或稳定的本地事件通道
- 浏览器插件上报 URL、标题、可交互元素和页面变化
- Yuki 可以请求页面扫描
- 执行子 Agent 使用元素 ID 执行点击、输入、滚动和导航
- 每个浏览器动作都返回结果并进入任务审计记录
- 网页操作和桌面操作使用相同的工具确认机制

不直接复用终端交互循环。原型中的命令行输入适合调试，不适合正式运行时。

### 阶段六：电脑操作 Agent

参考：

- `reference/YukiV6/core/maid.py`

目标是构建高能力但受运行时约束的电脑操作工具。

工具类别：

- 文件读取、目录浏览和文件搜索
- 进程与系统信息查询
- 浏览器页面观察和操作
- 截图与视觉分析
- 受限终端命令
- 脚本生成、运行和结果读取
- 技能保存、读取和复用
- 定时任务
- 可选的应用控制和窗口控制

建议保留 `maid.py` 中的有效设计：

- 任务循环
- 工具注册
- JSON 工具调用
- 临时工作区
- 输出截断
- 超时控制
- Windows 进程树终止
- 独立执行环境
- 任务追踪日志
- 技能复用

需要重写或加强的部分：

- 用显式权限策略替代关键词拦截
- 所有路径通过根目录和权限范围校验
- 将读、写、执行、网络、安装、删除分为不同能力级别
- 高风险操作必须等待用户确认
- 工具参数使用结构化 schema 校验
- 终端命令优先使用参数化进程调用，减少 shell 拼接
- 脚本执行使用隔离工作区和资源限制
- 禁止智能体自行提升权限
- 禁止默认执行支付、账号认证、系统破坏和不可逆删除
- 所有工具调用记录调用者、参数摘要、结果、耗时和确认记录

权限等级建议：

| 等级 | 能力 | 默认策略 |
|------|------|----------|
| L0 | 查询状态、读取公开信息 | 自动允许 |
| L1 | 读取用户指定目录、截图、网页扫描 | 自动允许或首次确认 |
| L2 | 创建或修改工作区文件、运行受限脚本 | 每次任务确认 |
| L3 | 修改项目文件、安装依赖、控制应用 | 明确确认 |
| L4 | 删除、发布、账号、支付、系统级操作 | 默认禁止，需人工解锁 |

“Omni 电脑操控能力”应理解为覆盖多种输入输出渠道和应用操作，而不是取消权限控制。

### 阶段七：状态中心和控制中心

工作内容：

- 增加 Yuki 当前状态显示
- 显示连接、推理、工具、观察、等待和错误状态
- 增加精力、活跃度、注意力、亲密度或其他数值模型
- 为数值计算建立可解释的状态来源和更新时间
- 增加桌面控制中心
- 支持 LLM、视觉、主动对话、窥屏、工具权限和待机动作开关
- 支持查看任务历史和审计日志
- 支持手动暂停所有主动行为
- 支持导出调试信息

控制中心必须区分：

- 当前状态
- 用户配置
- 运行时临时状态
- 工具权限
- 历史和审计记录

### 阶段八：音频增强

暂不作为当前主线，后续接入：

- GPT-SoVITS
- RVC
- 流式 TTS
- 语音输入和 VAD
- 音频播放队列
- 音频中断和恢复
- 口型同步增强

当前架构只需要提前保留：

```text
文本输出 → TTS Provider → Audio Event → Live2D Audio Player
```

不要在当前阶段引入 GPT-SoVITS 或 RVC 的具体依赖。

## 5. 模块复用策略

### 直接复用

- LiveYukiL2D 当前的 Live2D Cubism Core、Framework 和 Electron 壳层
- YukiV6 的 LLM 客户端可靠性设计
- YukiV6 的 Function Registry、Tool Context 和 Tool Call Manager 思路
- YukiV6 的 Maid 任务循环、临时工作区、技能复用和任务日志设计
- webpage_agent 的页面快照和动作队列协议思路

### 改造后复用

- YukiV6 的 `SessionPipeline`：改为单管线、单会话状态
- YukiV6 的 `YukiEngine`：拆为主 Agent、回复策略、上下文构建和工具调用
- YukiV6 的 `YukiState`：提炼为单实例状态机
- `maid.py`：改造成权限分级的执行 Agent
- webpage_agent：由 HTTP 轮询改为本地实时事件通道

### 暂不复用

- YukiV6 的 QQ/NapCat 监听和发送层
- Open-LLM-VTuber 的完整 React 前端
- Open-LLM-VTuber 的多客户端群聊管理
- 按 `chat_id` 的群聊会话模型
- GPT-SoVITS 和 RVC 运行时

## 7. 长期风险和约束

### 单管线阻塞

单管线简化了会话模型，但一个长时间工具任务可能阻塞聊天和主动行为。需要支持：

- 当前任务状态
- 可取消任务
- 后台执行子任务
- 任务完成事件回流
- 主聊天优先级

### 主 Agent 与执行 Agent 边界

执行 Agent 的结果必须回到主 Agent，由主 Agent 决定是否展示、如何表达和是否写入记忆。

### 云端视觉隐私

屏幕内容可能含有密码、聊天、文件和个人信息。必须将截图权限作为独立开关，并支持排除区域。

### 高权限工具安全

“更高权限”不能等同于“无限权限”。必须保留人工确认、操作范围、超时、审计和紧急停止。

### 数值系统过度复杂

精力、活跃度等数值应服务于可观察行为，先实现少量可解释指标，避免建立无法调试的隐藏人格状态。

## 8. 近期执行顺序

1. 保持当前 Live2D/Electron 启动链路稳定
2. 将前端输入和后端输出协议扩展为统一事件协议
3. 建立单管线 `RuntimeState` 和任务取消机制
4. 加入聊天框和文本 LLM 链路
5. 迁移 YukiV6 的 LLM 客户端、上下文构建和工具注册思想
6. 实现 Yuki 主 Agent 与执行子 Agent 的最小闭环
7. 增加状态显示和主动对话开关
8. 增加基础截图视觉理解，默认关闭主动窥屏
9. 将 webpage_agent 改造成浏览器观察适配器
10. 扩展受控电脑操作工具和审计系统
11. 最后接入复杂音频链路、数值控制中心和更多 Omni 能力

## 9. 完成标准

长期目标完成时，系统应满足：

- 一个 Yuki 实例稳定绑定一个 LLM 和一个 Live2D
- 聊天、主动对话和 Live2D 表现使用同一条主管线
- 用户可以随时暂停主动行为、视觉观察和电脑操作
- Yuki 可以调用执行子 Agent 完成多步骤电脑任务
- 高风险操作有明确确认，不依赖提示词自觉
- 屏幕和浏览器观察可追溯、可关闭、可限制范围
- 工具执行结果不会绕过 Yuki 直接对外输出
- 状态、数值、任务和权限可以在控制中心查看
- 后续接入 GPT-SoVITS、RVC 或新的视觉服务时不需要重写主管线
