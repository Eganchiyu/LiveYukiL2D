# LiveYukiL2D 项目公约与规则

## 项目定位

LiveYukiL2D 是一个单用户、单会话、单 LLM 和单 Live2D 模型绑定的桌面智能体项目。当前基础是本地 Python 服务、Vite 前端和 Yuki 模型，用于逐步实现文本聊天、主动行为、视觉理解、Live2D 表现和受控电脑操作。

项目保持单管线架构，不按 `chat_id` 拆分会话。GPT-SoVITS、RVC 等复杂音频链路属于后续增强，不得阻塞当前文本、状态和工具能力建设。

## 目录职责

- `server.py`：Python aiohttp 服务入口，负责静态资源托管、模型资源托管、WebSocket 广播和测试 API。
- `main.py`：项目启动薄入口，只调用 `server.main()`。
- `liveyuki_l2d/`：后端协议辅助代码，放置可复用的消息构造逻辑。
- `models/`：Live2D 模型资源目录，保持模型文件结构与 `.model3.json` 引用一致。
- `frontend/minimal/`：最小前端工程，负责 Live2D 渲染、模型加载、WebSocket 接收和浏览器调试接口。
- `frontend/vendor/`：外部参考或拷贝代码，除同步上游或明确需要外，不做业务修改。
- `docs/`：项目说明、文件说明、接口说明和维护文档。
- `references/`：两个相关完整项目参考代码、文档和资源。包括Live2D协议和Yuki代码实现。

## Python 规则

- Python 版本以 `.python-version` 为准，当前为 `3.13`；`pyproject.toml` 保持最低兼容 `>=3.10`。
- 依赖优先通过 `uv` 管理，修改 Python 依赖时同步更新 `pyproject.toml` 和 `uv.lock`。
- 服务端默认监听 `127.0.0.1:18765`，如需改动端口，应同步更新 README、前端默认模型地址和相关文档。
- WebSocket 消息结构应保持简单明确，新增消息类型时优先补充到 `liveyuki_l2d/protocol.py`。
- 对外输入边界需要做路径安全和资源存在性检查；内部调用不做过度防御。
- 服务端运行时应逐步围绕统一 `RuntimeState` 管理连接、用户活跃度、Yuki 状态、当前任务、视觉状态、桌宠状态和权限状态。
- 主管线顺序保持为：输入收集、状态更新、可选观察、上下文构建、Yuki 决策、工具调用或直接回复、输出事件、状态与记忆保存。
- 同一时刻只允许一个主推理任务运行；长任务必须支持状态查询、取消、后台执行和完成事件回流，不能无期限阻塞聊天和主动行为。

## 前端规则

- `frontend/minimal/` 是 Vite + TypeScript 最小工程，不引入 React 等额外框架，除非项目目标明确变化。
- Live2D SDK 相关代码尽量保持原始结构，业务适配优先放在 `frontend/minimal/src/main.ts` 或独立适配层。
- 前端默认连接本地服务 `127.0.0.1:18765`，不要随意改为公网地址。
- 构建产物位于 `frontend/minimal/dist/`，服务端当前从该目录托管页面。
- 前端与运行时之间优先使用统一事件协议，事件至少覆盖文本、音频、表情、动作、状态、工具调用、观察结果和错误。
- 聊天、主动对话、工具执行和 Live2D 表现应共享同一运行时状态，不在前端单独维护互相冲突的业务状态。

## Live2D 模型资源规则

- `models/Yuki/` 中的 `.model3.json`、`.moc3`、贴图、物理文件必须保持相对路径可用。
- 当前 Yuki 模型没有 `.exp3.json` 和 `.motion3.json`，不要假设表情或动作一定有效。
- 调整模型缩放、初始位置、交互配置时，优先修改服务端 `YUKI_MODEL_INFO`，并同步说明文档。

## Agent 与运行时边界

- Yuki 是唯一对外人格和最终决策者，负责理解输入、决定是否回复或主动观察、选择工具、表达执行结果以及决定 Live2D 表现。
- 执行子 Agent 只负责任务分解、信息查询、工具调用和结构化汇报，不直接向用户发送消息，不绕过 Yuki 写入对外事件或长期记忆。
- 执行子 Agent 可以接收有限的人格参数，但权限、工具白名单、确认策略、超时和资源限制必须由运行时强制控制，不能由提示词决定。
- 主 Agent、执行子 Agent、LLM、视觉服务和工具注册中心应保持适配层边界，优先复用成熟模块，不复制完整外部应用。

## 工具与安全规则

- 工具能力必须通过注册中心和结构化参数校验提供，禁止依赖提示词自觉或关键词拦截作为唯一安全措施。
- 权限至少区分四类边界：读取、写入、执行、网络；路径必须经过允许根目录和权限范围校验。
- 权限等级遵循 L0-L4：公开查询自动允许；指定目录读取、截图和网页扫描可自动允许或首次确认；工作区修改和受限脚本每次任务确认；项目修改、依赖安装和应用控制明确确认；删除、发布、账号、支付和系统级操作默认禁止。
- 高风险、不可逆、涉及账号认证、支付、系统破坏或权限提升的操作默认禁止，不能由智能体自行解锁或提升权限。
- 工具调用应记录调用者、参数摘要、权限等级、确认记录、状态、结果摘要、耗时和审计标识；执行失败也必须形成可追踪事件。
- 视觉观察默认关闭，必须支持明确启用、暂停和立即停止；上传云端前应限制截图范围，排除密码框和隐私区域，API Key 只能从环境变量或本地安全配置读取。

## 状态与协议规则

- 逐步统一 `AppState`、`ConversationMessage`、`AgentEvent` 和 `ToolExecution` 数据模型，新增字段应说明来源、更新时间和生命周期。
- 工具结果必须回到主 Agent，由主 Agent 决定是否展示、如何表达以及是否写入记忆。
- 新协议应优先采用结构化事件并保留错误和状态事件；原有 `/api/say`、`/api/audio` 和 `/ws` 接口在没有迁移说明前保持兼容。
- 主动对话必须具备总开关、安静时段、最短间隔、用户活跃度判断、明确拒绝后的冷却时间和失败退避策略。
- 数值状态只实现可解释且能驱动可观察行为的指标，避免引入无法调试的隐藏人格状态。

## 文档规则

- 根目录文件职责说明维护在 `docs/root-files.md`。
- 修改启动方式、端口、API、目录职责或模型资源结构时，同步更新 README 和 docs 中对应文档。
- 文档内容以中文为主，命令和路径保持可复制。
- 修改运行时架构、协议、权限、视觉观察、Agent 边界或开发阶段目标时，同步更新 `docs/development-plan.md` 或相关设计文档。
- 阶段性实现应优先保持现有 Live2D、Electron 和 minimal 前端启动链路稳定；暂不为迁移群聊、多 `chat_id` 会话或完整 React 应用引入额外复杂度。

## Git 与生成文件规则

- 不提交 `.venv/`、`__pycache__/`、构建缓存等本地生成文件。
- 若决定提交前端构建产物，需要明确它是服务端运行所依赖的静态页面，并避免混入无关产物。
- 不在仓库中提交私钥、令牌、个人配置、临时音频缓存或大体积无关文件。

## 修改原则

- 优先做最小必要修改，不做与当前需求无关的重构。
- 修改现有行为前先阅读相关文件，确认调用链和端口、路径依赖。
- 外部拷贝代码和 SDK 文件谨慎改动，必要时在项目适配层解决。
